# 09.8 — Corriger fhir.service.ts : ne pas bloquer l'export si pas de vitals

**Réf** : `260426.md:672` (LB14)  
**Priorité** : 🟡 MANQUANT — CONFORMITÉ GDPR/DPDPA  
**Durée estimée** : 10 min  
**Dépend de** : P1

---

## Contexte

`fhir.service.ts` bloque l'export FHIR si le patient n'a pas de vitals ou de visites. Ceci **viole le droit à la portabilité** (GDPR Art.20 / DPDPA). Le patient a droit à TOUTES ses données, même partielles.

---

## Fichier concerné

```
apps/api/src/app/fhir/fhir.service.ts:53-69
```

---

## Action

**Avant** (bloque si pas de données) :
```ts
if (!vitals.length && !visits.length) {
  throw new NotFoundException('No data to export');
}
```

**Après** (exporte ce qui existe) :
```ts
// Toujours retourner un Bundle FHIR, même vide
const bundle: FHIR.Bundle = {
  resourceType: 'Bundle',
  type: 'searchset',
  total: 0,
  entry: [],
};

if (vitals.length > 0) {
  bundle.entry!.push(...vitals.map(mapToFhirObservation));
  bundle.total! += vitals.length;
}
if (visits.length > 0) {
  bundle.entry!.push(...visits.map(mapToFhirEncounter));
  bundle.total! += visits.length;
}

return bundle;
```

## Vérification

```bash
# Patient sans données → retourne Bundle vide (200) au lieu de 404
curl http://localhost:3000/api/fhir/Patient/empty-patient/\$export
# Doit retourner 200 avec un Bundle de total=0
```

## Risques

- **Moyen** : Les clients FHIR peuvent ne pas gérer un Bundle vide → vérifier la spec FHIR R4
