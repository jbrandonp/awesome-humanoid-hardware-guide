# 05.4 — Corriger dpdpa-consent : scope de révocation

**Réf** : `260426.md:70` (A7) + `260426.md:560` (INT1) + `260426.md:1094-1096`  
**Priorité** : 🟠 P5 — CRITIQUE INTÉGRITÉ DONNÉES  
**Durée estimée** : 20 min  
**Dépend de** : P1

---

## Contexte

`revokeConsentAndPurge()` supprime **TOUTES** les données d'un patient quand **UN SEUL** médecin révoque son consentement. Si le patient a plusieurs médecins, les données liées aux autres médecins sont détruites → **catastrophe multi-provider**.

---

## Fichier concerné

```
apps/api/src/app/dpdpa/dpdpa-consent.service.ts:127-246
```

---

## Action

Modifier les requêtes de suppression pour filtrer par `userId` :

**Avant** (supprime tout) :
```ts
await this.prisma.vital.deleteMany({ where: { patientId } });
await this.prisma.prescription.deleteMany({ where: { patientId } });
```

**Après** (supprime seulement celles du médecin) :
```ts
await this.prisma.vital.deleteMany({
  where: { patientId, recordedBy: userId }
});
await this.prisma.prescription.deleteMany({
  where: { patientId, prescriberId: userId }
});
```

---

## Vérification

```bash
# Test : 2 médecins, 1 révoque → seulement ses données supprimées
# Les données de l'autre médecin restent intactes
```

---

## Risques

- **Élevé** : Si les données n'ont pas de champ `recordedBy`/`prescriberId` → le filtre est impossible
- Alternative : ajouter une table de jointure `UserDataAccess` pour tracker qui a accès à quoi
