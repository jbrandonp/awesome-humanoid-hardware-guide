# 05.3 — Corriger emarsync : Redis.set avant décrément DB

**Réf** : `260426.md:71` (A8) + `260426.md:605` (AB5-AB6) + `260426.md:1090-1092`  
**Priorité** : 🟠 P5 — CRITIQUE INTÉGRITÉ DONNÉES  
**Durée estimée** : 15 min  
**Dépend de** : P1

---

## Contexte

Dans `emarsync-listener.interceptor.ts`, le stock est décrémenté en DB **avant** le `Redis.set`. Si Redis échoue après le décrément DB → **incohérence** : stock baissé mais pas d'idempotence → double-déduction au retry.

---

## Fichier concerné

```
apps/api/src/app/emar/emarsync-listener.interceptor.ts:78-103
```

---

## Action

**Avant** (DB d'abord) :
```ts
// Décrement DB
await this.prisma.stock.update({ where: { id }, data: { quantity: { decrement: qty } } });
// Puis Redis
await this.redis.set(idempotencyKey, '1');
```

**Après** (Redis d'abord avec NX) :
```ts
// Redis SET NX (atomique, échoue si déjà présent)
const acquired = await this.redis.set(idempotencyKey, '1', 'NX', 'EX', 30);
if (!acquired) throw new ConflictException('Requête déjà traitée');

// Puis décrément DB
await this.prisma.stock.update({ where: { id }, data: { quantity: { decrement: qty } } });
```

---

## Vérification

```bash
# Test de concurrence : 2 requêtes simultanées → 1 seul décrément
```

---

## Risques

- **Moyen** : Si la DB échoue après le Redis.set → stock pas décrémenté mais idempotence consommée
- Solution : ajouter un rollback Redis en cas d'échec DB
