# 05.2 — Déplacer l'idempotence billing dans la transaction

**Réf** : `260426.md:68` (A5) + `260426.md:1087-1088`  
**Priorité** : 🟠 P5 — CRITIQUE INTÉGRITÉ DONNÉES  
**Durée estimée** : 15 min  
**Dépend de** : P1

---

## Contexte

La vérification d'idempotence dans `billing.service.ts` est faite **hors transaction**. Entre le check et l'insertion, une requête concurrente peut créer une facture → **factures en double**.

---

## Fichier concerné

```
apps/api/src/app/billing/billing.service.ts:50-63
```

---

## Action

**Avant** (check hors transaction) :
```ts
// Check idempotence
const existing = await this.prisma.bill.findFirst({ where: { idempotencyKey } });
if (existing) return existing;

// Insertion
const bill = await this.prisma.bill.create({ data: { ... } });
```

**Après** (check dans transaction) :
```ts
const bill = await this.prisma.$transaction(async (tx) => {
  const existing = await tx.bill.findFirst({ where: { idempotencyKey } });
  if (existing) return existing;

  return tx.bill.create({ data: { ... } });
});
```

---

## Vérification

```bash
# Test de concurrence : 2 requêtes simultanées avec même idempotencyKey
# → Une seule facture créée
```

---

## Risques

- **Moyen** : La transaction peut échouer si isolation level insuffisant
- Alternative : utiliser une contrainte UNIQUE sur `idempotencyKey` en base + `ON CONFLICT`
